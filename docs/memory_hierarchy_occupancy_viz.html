GPU Memory Hierarchy & Occupancy

Why gemvWarpSharedXKernel (manually caching x in shared memory) ran ~20% slower than the plain warp-per-row kernel — measured on a GTX 1060 Max-Q (GP106, sm_61). Two stacked reasons: L2 was already caching x for free, and the 16 KB/block shared allocation crashed occupancy. Step through the story, then play with the occupancy calculator below.

Each step down is roughly 10× slower and ~10× bigger. Caches (L1, L2) are automatic — hardware decides what stays hot. Shared memory is manual — same speed as L1, but you do the filling, you pay the barrier, and you rent the space from the SM's 96 KB budget.

The decisive line: the staging loop's global reads of x hit L2 anyway, so shared-x saves zero DRAM traffic — it can only add costs. Shared memory pays off when it saves traffic L2 can't absorb (matmul tiles, convolution halos), not for a 16 KB vector sitting comfortably inside 1.5 MB of L2.

Occupancy calculator — how shared memory rents out your latency hiding

An SM hides DRAM latency by keeping many blocks resident and switching between their warps: while one warp waits ~500 cycles on a load, others run. Staging x in shared memory re-implemented by hand what L2 was already doing for free (x = 16 KB ≪ 1.5 MB L2, re-read by all 4096 rows → permanent L2 resident after the first touch). It saved no DRAM traffic, and it paid twice: (1) 4096 extra loads + 4096 shared stores + a __syncthreads() in every one of the 1024 blocks, and (2) the 16 KB/block allocation let only ⌊96/16⌋ = 6 blocks fit per SM instead of the 16 the thread budget allows — 24 resident warps instead of 64, so the SM had far fewer warps to switch to while A's loads stalled. Rule of thumb: spend shared memory only to save traffic the caches can't — heavy block-local reuse of data too big or too thrashed for L2 (matmul tiles, convolution halos) — never to duplicate a small read-only vector that already fits in L2.
