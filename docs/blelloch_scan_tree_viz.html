Blelloch Work-Efficient Scan — The Implicit Tree

Scaled-down model: n = 8 elements (the real kernel does 1024 per block, two elements per thread). The array along the bottom is the live __shared__ memory s[]; everything above it is the implicit binary tree the algorithm walks — not stored separately, just an overlay on the same 8 slots. Notice every internal node sits directly above the rightmost index of the range it covers: that is where its sum physically lives, overwriting the leaf that was there.

// Up-sweep (reduce): build partial sums up the tree for (d = n>>1; d > 0; d >>= 1) if (tid < d) s[bi] += s[ai]; s[n-1] = 0; // clear the root → seeds the exclusive scan // Down-sweep: push prefixes back down the tree for (d = 1; d < n; d <<= 1) if (tid < d) { t=s[ai]; s[ai]=s[bi]; s[bi]+=t; } // ai = offset*(2*tid+1)-1 (left child) , bi = offset*(2*tid+2)-1 (right child)

How the tree is involved — in detail

There is no separate tree in memory. The 8-slot array is the tree, the way a segment tree or a heap is an array: a complete binary reduction tree laid over the indices. Two facts define the layout:

  • Leaves = the input elements in natural left-to-right order (slots 0…7).
  • Each internal node lives at the index of the rightmost leaf in its range. That is the entire meaning of the -1 in bi = offset*(2*tid+2)-1: it always lands on the right child, so sums pile up rightward.

Up-sweep builds the tree (and overwrites in place)

Each level halves the number of active nodes and writes each parent's sum onto its right child's slot. Watch slot 7: it is reused at every level, because many ranges share the same rightmost index. That stacking — one node sitting on top of another at the same column — is the in-place overwrite, and it is why the layout is not a pre/in/post-order serialization (those need a distinct slot per node).

slot 7 after…holds the sum of range
load[7]  (just the leaf)
level 1 (offset 1)[6–7]
level 2 (offset 2)[4–7]
level 3 (offset 4)[0–7]  = grand total

So after the up-sweep the root (whole-array total) sits at the last slot, s[n-1]. That is why the down-sweep seeds itself by clearing exactly that slot.

Down-sweep drains the tree

Set the root to 0, then walk back down. At every node a thread does the swap-and-add t = s[ai]; s[ai] = s[bi]; s[bi] += t;. Read it as: the left child takes the parent's incoming prefix; the right child takes that same prefix plus the left child's old subtree sum. Those subtree sums are exactly the values the up-sweep parked in the tree, which is why both sweeps use the identical ai/bi addressing — the down-sweep is just reading back what the up-sweep stored. When you reach the leaves, each slot holds the sum of everything strictly to its left: the exclusive scan.

Why bother — the work argument

The tree is what buys work efficiency. Up-sweep does n-1 adds, down-sweep does n-1 adds and swaps: O(n) total, versus Hillis–Steele's O(n log n). Depth is 2·log&sub2;n either way. The catch, and the thing to measure on the GTX 1060: the 2^d strides make consecutive threads hit the same shared-memory bank, so the theoretical work win can be eaten by bank conflicts (see bank_conflict_viz.html) unless you pad the indices.
