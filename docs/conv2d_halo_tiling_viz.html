Why it works — and the four ways it bites

The two mapping formulas (all the indexing there is)

Loading — treat the 10×10 tile as a flat list of 100 slots. Thread t = ty*6+tx fills slots s = t, t+36, t+72 (a loop: for (s = t; s < 100; s += 36)). Each slot unflattens to tile row s / 10, column s % 10 (integer divide and remainder), which is global pixel (blockIdx.y*6 − 2 + s/10,  blockIdx.x*6 − 2 + s%10) — outside the image → store 0.0f. Consecutive slots are consecutive columns, so the global reads coalesce along rows.

Computing — thread (ty,tx)'s window is simply tile rows ty..ty+2r, columns tx..tx+2r: because the tile starts r pixels up-and-left of the output tile, the window's top-left corner lands exactly at (ty,tx). No bounds check anywhere — the zeros are already in the tile.

The four ways it bites

  • The stride bug (toggle it above!). The tile's row stride is the tile side (10 here, 24 for real), NOT the block width. Address with the wrong stride and the window rows collapse onto each other and wrap — plausible-looking, wrong everywhere. This is what the odd-size correctness cases catch.
  • Barrier before any read. Slot 47 is loaded by thread 11, but read by many other threads' windows. __syncthreads() between the load loop and the compute loop — no exceptions.
  • No early return. A thread whose output pixel is outside the image (right/bottom edge blocks) must STILL load its share of slots and STILL reach the barrier. Guard only the final out[y*W+x] = sum write.
  • Every slot gets written. The zero-fill else-branch is not optional: shared memory starts as garbage, and an unwritten halo slot is a silent wrong answer.
